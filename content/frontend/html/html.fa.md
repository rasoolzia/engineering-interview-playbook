## سوال 1

**آیدی**: html-001  
**عنوان**: تفاوت `<div>` و `<span>` چیست؟  
**سطح دشواری**: آسان  
**دسته‌بندی**: اصول اولیه

### جواب

`<div>` و `<span>` هر دو عناصر عمومی (Generic) هستند، اما در نحوه نمایش و کاربرد تفاوت دارند.

**`<div>`**

- عنصر block-level
- کل عرض موجود را اشغال می‌کند
- از خط جدید شروع می‌شود
- برای گروه‌بندی بخش‌های بزرگ‌تر استفاده می‌شود

**`<span>`**

- عنصر inline
- فقط به اندازه محتوای خود عرض می‌گیرد
- از خط جدید شروع نمی‌شود
- برای بخش‌های کوچک داخل متن استفاده می‌شود

**آیدی**: html-002

**سطح دشواری**: آسان

### جواب

اعلان `<!DOCTYPE>` به مرورگر اعلام می‌کند که سند با چه نسخه‌ای از HTML نوشته شده است.

اهداف اصلی آن:

- جلوگیری از فعال شدن حالت سازگاری قدیمی (Quirks Mode)

## سوال 3

**آیدی**: html-003  
**عنوان**: تفاوت عناصر Block و Inline چیست؟  
**سطح دشواری**: آسان  
**دسته‌بندی**: اصول اولیه

### جواب

تفاوت این عناصر در نحوه نمایش آن‌ها در صفحه است.

**عناصر Block**

- از خط جدید شروع می‌شوند
- کل عرض موجود را اشغال می‌کنند
- می‌توانند شامل عناصر Block یا Inline باشند

مثال: `<div>`، `<p>`، `<section>`

**عناصر Inline**

- از خط جدید شروع نمی‌شوند
- فقط به اندازه محتوای خود عرض می‌گیرند
- معمولاً داخل متن استفاده می‌شوند

مثال: `<span>`، `<a>`، `<strong>`

## سوال 5

**آیدی**: html-005  
**عنوان**: تفاوت `id` و `class` چیست؟  
**سطح دشواری**: آسان  
**دسته‌بندی**: اصول اولیه

### جواب

`id` و `class` هر دو برای شناسایی عناصر استفاده می‌شوند اما تفاوت‌هایی دارند.

**`id`**

- در هر صفحه باید یکتا باشد
- برای هدف‌گیری یک عنصر خاص استفاده می‌شود
- معمولاً در JavaScript یا لینک‌های لنگر (anchor) کاربرد دارد

**`class`**

- می‌تواند روی چندین عنصر استفاده شود
- برای گروه‌بندی عناصر با استایل مشابه استفاده می‌شود
- رایج‌ترین روش هدف‌گیری در CSS است

هر عنصر می‌تواند چندین class داشته باشد اما فقط یک id دارد.

## 🧠 سوال ۶

**آیدی**: html-006  
**عنوان**: هدف از ویژگی `alt` در تگ‌های `<img>` چیست؟  
**سطح دشواری**: آسان  
**دسته‌بندی**: معنایی و دسترس‌پذیری

### پاسخ 📄

ویژگی `alt` متنی جایگزین برای تصویر فراهم می‌کند زمانی که تصویر قابل نمایش نباشد یا زمانی که از فناوری‌های کمکی استفاده می‌شود.

اهداف کلیدی:

- **دسترس‌پذیری**: صفحه‌خوان‌ها متن alt را برای کاربران کم‌بینا می‌خوانند
- **سئو**: موتورهای جستجو از متن alt برای درک محتوای تصویر استفاده می‌کنند
- **بازگشتی**: اگر تصویر بارگذاری نشود، متن نمایش داده می‌شود

بهترین روش:

- توصیفی اما مختصر باشد
- برای تصاویر تزئینی از `alt=""` استفاده کنید

مثال:

```html
<img src="profile.jpg" alt="Portrait of Jane Doe smiling" />
```

## 🧠 سوال ۷

**آیدی**: html-007  
**عنوان**: عناصر معنایی HTML مانند `<article>`، `<section>`، `<aside>` را توضیح دهید و اینکه چه زمانی به جای `<div>` از آنها استفاده کنیم.  
**سطح دشواری**: متوسط  
**دسته‌بندی**: معنایی و دسترس‌پذیری

### پاسخ 📄

عناصر معنایی معنای خود را به وضوح برای مرورگرها، توسعه‌دهندگان و فناوری‌های کمکی توصیف می‌کنند.

- `<article>`: محتوای مستقلی که می‌تواند به تنهایی قابل استفاده باشد (مثلاً پست وبلاگ، خبر)
- `<section>`: گروه‌بندی موضوعی محتوا (مثلاً فصل، مقدمه)
- `<aside>`: محتوایی که به طور غیرمستقیم با محتوای اصلی مرتبط است (مثلاً نوار کناری، لینک‌های مرتبط، تبلیغات)

مزایا نسبت به `<div>`:

- دسترس‌پذیری بهتر (صفحه‌خوان‌ها نقش‌ها را اعلام می‌کنند)
- سئوی بهبود یافته (موتورهای جستجو ساختار را بهتر درک می‌کنند)
- کد قابل نگهداری‌تر

مثال:

```html
<article>
  <h2>جاوااسکریپت چگونه کار می‌کند</h2>
  <p>یک پست مستقل درباره درونیات JS...</p>
</article>

<section>
  <h2>خدمات ما</h2>
  <p>محتوای موضوعی گروه‌بندی‌شده...</p>
</section>

<aside>
  <h3>مقالات مرتبط</h3>
  <ul>
    <li><a href="#">بررسی عمیق closure‌ها</a></li>
  </ul>
</aside>
```

## 🧠 سوال ۸

**آیدی**: html-008  
**عنوان**: تفاوت‌های بین `<input type="number">` و `<input type="tel">` چیست؟  
**سطح دشواری**: متوسط  
**دسته‌بندی**: فرم‌ها

### پاسخ 📄

هر دو نوع ورودی برای ورودی‌های عددی مانند استفاده می‌شوند، اما رفتار متفاوتی دارند.

**`type="number"`**

- فقط مقادیر عددی را می‌پذیرد
- اعتبارسنجی داخلی دارد
- در برخی مرورگرها فلش‌های افزایش/کاهش را نمایش می‌دهد
- بسته به تنظیمات ممکن است کاراکترهایی مانند "+" یا "-" را رد کند

**`type="tel"`**

- برای شماره تلفن طراحی شده
- اعتبارسنجی عددی اعمال نمی‌کند
- صفحه کلید عددی را در دستگاه‌های موبایل فعال می‌کند
- امکان قالب‌بندی انعطاف‌پذیر را فراهم می‌کند (مثلاً +۹۸-۲۱-۱۲۳۴-۵۶۷۸)

برای شماره تلفن از `type="tel"` و برای داده‌های عددی واقعی از `type="number"` استفاده کنید.

مثال:

```html
<!-- فقط عدد می‌پذیرد و فلش افزایش/کاهش نمایش می‌دهد -->
<input type="number" min="1" max="100" value="10" />

<!-- صفحه‌کلید عددی موبایل را فعال می‌کند، فرمت انعطاف‌پذیر -->
<input type="tel" placeholder="+98-21-1234-5678" />
```

## 🧠 سوال ۹

**آیدی**: html-009  
**عنوان**: چگونه می‌توان عملکرد صفحه را با استفاده از ویژگی‌ها یا تکنیک‌های HTML بهبود بخشید؟  
**سطح دشواری**: متوسط  
**دسته‌بندی**: عملکرد

### پاسخ 📄

چندین ویژگی HTML به بهینه‌سازی بارگذاری و رندر کمک می‌کنند:

- استفاده از `loading="lazy"` روی `<img>` و `<iframe>` برای بارگذاری تأخیری
- اضافه کردن `async` یا `defer` به تگ‌های `<script>` برای جلوگیری از مسدود شدن رندر
- استفاده از `<link rel="preload">`، `preconnect`، یا `dns-prefetch` برای منابع حیاتی
- مشخص کردن `width` و `height` برای تصاویر جهت جلوگیری از جابجایی طرح (CLS)
- استفاده از تصاویر واکنش‌گرا با `srcset` و `<picture>`

این تکنیک‌ها زمان بارگذاری را بهبود می‌بخشند، مسدودیت را کاهش می‌دهند و تجربه کاربری را ارتقا می‌دهند.

مثال:

```html
<img src="large.jpg" loading="lazy" width="800" height="600" alt="..." />
<script src="app.js" defer></script>
```

## 🧠 سوال ۱۰

**آیدی**: html-010  
**عنوان**: تگ `<meta name="viewport">` چیست و چرا برای طراحی واکنش‌گرا مهم است؟  
**سطح دشواری**: آسان  
**دسته‌بندی**: رفتار مرورگر

### پاسخ 📄

تگ متا ویوپورت نحوه نمایش صفحه وب را در دستگاه‌های موبایل کنترل می‌کند.  
بدون آن، مرورگرهای موبایل اغلب صفحات را با عرض دسکتاپ رندر کرده و سپس آنها را کوچک می‌کنند.

استفاده استاندارد:

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

- `width=device-width`: عرض صفحه را با عرض صفحه‌نمایش دستگاه تطبیق می‌دهد
- `initial-scale=1.0`: سطح زوم اولیه را تنظیم می‌کند

این تگ برای طراحی واکنش‌گرا ضروری است.

## 🧠 سوال ۱۱

**آیدی**: html-011  
**عنوان**: تفاوت ویژگی‌های `async` و `defer` در تگ‌های `<script>` چیست؟ تاثیر آن‌ها روی Critical Rendering Path چیست؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: عملکرد

### پاسخ 📄

هر دو ویژگی `async` و `defer` بارگذاری غیرمسدودکننده اسکریپت را ممکن می‌کنند، اما زمان اجرای آن‌ها متفاوت است:

- **async**:
  - دانلود اسکریپت به صورت موازی انجام می‌شود
  - به‌محض اتمام دانلود اجرا می‌شود (ممکن است تجزیه HTML را قطع کند)
  - ترتیب اجرا بین چند اسکریپت async تضمین نمی‌شود
  - مناسب اسکریپت‌های مستقل مثل آنالیتیکس و تبلیغات

- **defer**:
  - دانلود اسکریپت به صورت موازی انجام می‌شود
  - فقط بعد از تکمیل تجزیه HTML اجرا می‌شود (کمی قبل از DOMContentLoaded)
  - ترتیب اسکریپت‌ها حفظ می‌شود
  - برای اسکریپت‌هایی که به DOM وابسته‌اند مناسب‌تر است

**تاثیر روی Critical Rendering Path (CRP):**

- بدون این ویژگی‌ها → اسکریپت مسدودکننده رندر است (تجزیه متوقف می‌شود)
- `async` → ممکن است در صورت اجرای زودهنگام هنوز به رندر ضربه بزند
- `defer` → برای رندر عملاً غیرمسدودکننده است

بهترین روش: در اکثر سناریوها از `defer` استفاده کنید مگر اینکه اجرای فوری لازم باشد.

مثال:

```html
<script src="analytics.js" async></script>
<script src="app.js" defer></script>
```

## 🧠 سوال ۱۲

**آیدی**: html-012  
**عنوان**: تفاوت بین تگهای `<meta>` و هدرهای HTTP چیست و چه زمانی باید از هر کدام استفاده کرد؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: سئو و متا

### پاسخ 📄

هم تگهای `<meta>` و هم هدرهای HTTP متادیتا درباره یک سند ارائه میدهند، اما در سطوح مختلفی عمل میکنند.

### تگهای `<meta>`

- داخل بخش `<head>` سند HTML تعریف میشوند
- متادیتایی ارائه میدهند که توسط مرورگرها و موتورهای جستجو تفسیر میشود
- پس از شروع بارگذاری سند HTML پردازش میشوند

مثالهای رایج:

```html
<meta charset="UTF-8" />
<meta name="description" content="سوالات مصاحبه HTML" />
<meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

موارد استفاده:

- توضیحات صفحه برای سئو
- کدگذاری کاراکتر
- پیکربندی ویوپورت

### هدرهای HTTP

- توسط سرور قبل از سند HTML ارسال میشوند
- رفتار مرورگر را در سطح پروتکل کنترل میکنند
- معتبرتر از تگهای متا هستند

مثالها:

- `Content-Type`
- `Cache-Control`
- `Content-Security-Policy`
- `Set-Cookie`

موارد استفاده:

- سیاستهای امنیتی
- کنترل کش
- ریدایرکت
- مذاکره محتوا

## 🧠 سوال ۱۳

**آیدی**: html-013  
**عنوان**: چند اشتباه رایج در استفاده از ARIA چیست و چرا باید HTML بومی را نسبت به ARIA در اولویت گذاشت؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: معنایی و دسترسی‌پذیری

### پاسخ 📄

اشتباهات رایج:

1. استفاده از ARIA وقتی معادل بومی HTML وجود دارد (مثل `role="button"` روی `<div>` به جای `<button>`)
2. افزودن ARIA اضافی و تکراری (مثلاً `aria-label` برای تصویری که `alt` مناسب دارد)
3. فراموش کردن `aria-hidden` برای محتوای تزئینی
4. تنظیم نادرست live regionها (`aria-live="polite"` در مقابل `assertive`)
5. مدیریت نکردن درست فوکوس کیبورد در کامپوننت‌های سفارشی

**قانون طلایی (ARIA Authoring Practices):**  
**"No ARIA is better than bad ARIA"**  
**"If you can use a native semantic element, do it"**

ترتیب اولویت:

1. HTML بومی (مثل button، input، nav و ...)
2. نقش‌ها/حالت‌های ARIA فقط در صورت نیاز
3. JavaScript برای رفتارهای پویا

مثال بد → خوب:

```html
<!-- Bad -->
<div role="button" onclick="...">Click</div>

<!-- Good -->
<button onclick="...">Click</button>
```

از ابزارهایی مثل Lighthouse یا axe برای شناسایی خطاهای دسترسی‌پذیری استفاده کنید.

## 🧠 سوال ۱۴

**آیدی**: html-014  
**عنوان**: مسیر رندر بحرانی (Critical Rendering Path) چیست و HTML چه نقشی در آن دارد؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: عملکرد

### پاسخ 📄

مسیر رندر بحرانی (CRP) توالی مراحلی است که مرورگر برای رندر کردن یک صفحه طی میکند.

مراحل اصلی:

1. تجزیه HTML → ساخت DOM
2. تجزیه CSS → ساخت CSSOM
3. ترکیب در درخت رندر
4. چیدمان (Layout)
5. رنگ آمیزی (Paint)

HTML مستقیماً بر موارد زیر تأثیر میگذارد:

- اندازه و عمق DOM
- ترتیب بارگذاری منابع
- اسکریپت های مسدودکننده
- تصاویر بدون ابعاد مشخص

تکنیک های بهینه سازی HTML:

- استفاده از `defer` برای اسکریپت ها
- کمینه کردن عمق DOM
- مشخص کردن `width` و `height` برای تصاویر
- پیش بارگذاری منابع حیاتی

درک CRP برای بهینهسازی عملکرد ضروری است.

## 🧠 سوال ۱۵

**آیدی**: html-015  
**عنوان**: تفاوت بین ویژگیهای `id` و `data-*` در HTML چیست و چه زمانی باید از هر کدام استفاده کرد؟  
**سطح دشواری**: متوسط  
**دسته‌بندی**: اصول اولیه

### پاسخ 📄

هر دو ویژگی `id` و `data-*` اطلاعات را به عناصر متصل میکنند، اما اهداف متفاوتی دارند.

### `id`

- باید در سند یکتا باشد
- برای موارد زیر استفاده میشود:
  - ناوبری قطعه ای (`#section`)
  - انتخاب DOM در JavaScript
  - ارتباط برچسب ها (ویژگی `for`)
- معنای معنایی در ساختار سند دارد

مثال:

```html
<section id="pricing"></section>
```

### ویژگیهای `data-*`

- ویژگیهای سفارشی برای ذخیره داده های اضافی
- بر معناشناسی تأثیر نمیگذارند
- از طریق JavaScript قابل دسترسی هستند (`element.dataset`)

مثال:

```html
<button data-user-id="42"></button>
```

از `id` برای هویت سند استفاده کنید.  
از `data-*` برای ذخیره متادیتای سفارشی استفاده کنید.

## 🧠 سوال ۱۶

**آیدی**: html-016  
**عنوان**: تفاوت بین `<section>` و `<div>` از منظر طرح کلی سند چیست؟  
**سطح دشواری**: متوسط  
**دسته‌بندی**: معنایی و دسترسی‌پذیری

### پاسخ 📄

`<div>` یک کانتینر عمومی بدون معنای معنایی است.

`<section>` نشاندهنده گروهبندی موضوعی محتوا است و به طرح کلی سند کمک میکند.

تفاوتهای کلیدی:

- `<section>` معمولاً باید شامل یک عنوان باشد
- ساختار را به فناوریهای کمکی و موتورهای جستجو اعلام میکند
- `<div>` صرفاً ساختاری است و ارزش معنایی ندارد

از `<section>` زمانی استفاده کنید که محتوا بخش منطقی از سند را نشان می‌دهد.
از `<div>` فقط زمانی استفاده کنید که هیچ عنصر معنایی مناسبی وجود ندارد.

مثال:

```html
<!-- div: ظرف خالص برای layout، بدون نقش معنایی -->
<div class="card-wrapper">
  <div class="card">...</div>
</div>

<!-- section: بخش موضوعی سند، باید عنوان داشته باشد -->
<section>
  <h2>خدمات ما</h2>
  <p>ما توسعه وب، طراحی و مشاوره ارائه می‌دهیم.</p>
</section>
```

## 🧠 سوال ۱۷

**آیدی**: html-017  
**عنوان**: `<template>` و Shadow DOM در Web Components چگونه با هم کار می‌کنند و چه مسائلی را حل می‌کنند؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: رفتار مرورگر و HTML مدرن

### پاسخ 📄

`<template>` و Shadow DOM دو بخش اصلی Web Components هستند:

- `<template>`: قطعه HTML که parse می‌شود اما inert است (رندر یا side effect ندارد تا زمانی که clone شود)
- Shadow DOM: زیردرخت DOM کپسوله‌شده که به یک عنصر سفارشی متصل می‌شود (استایل و DOM ایزوله می‌ماند)

این دو در کنار هم این مشکلات را حل می‌کنند:

- نشت استایل (CSS کامپوننت روی کل صفحه اثر نمی‌گذارد)
- تداخل در انتخاب DOM (جلوگیری از conflict در `querySelector`)
- قابلیت استفاده مجدد بدون وابستگی به فریمورک (مثل React/Vue)

نمونه پایه:

```html
<template id="my-card">
  <style>
    .card {
      border: 1px solid;
      padding: 1em;
    }
  </style>
  <div class="card"><slot name="title"></slot></div>
</template>

<script>
  class MyCard extends HTMLElement {
    constructor() {
      super();
      const shadow = this.attachShadow({ mode: 'open' });
      const template = document
        .getElementById('my-card')
        .content.cloneNode(true);
      shadow.appendChild(template);
    }
  }
  customElements.define('my-card', MyCard);
</script>

<my-card><span slot="title">Hello</span></my-card>
```

مزیت‌ها: مستقل از فریمورک، کارآمد، و آینده‌نگر.

- ناوبری
- اسکریپت‌ها
- فرم‌ها

مثال (جاسازی امن):

```html
<iframe
  src="https://example.com/widget"
  sandbox="allow-scripts allow-same-origin"
></iframe>
```

بدون `sandbox` احتمال ریسک‌هایی مثل XSS یا clickjacking افزایش پیدا می‌کند.

- `srcset`: فهرستی از گزینه‌های تصویر (با descriptorهایی مثل 1x، 2x یا width)
- `sizes`: عرض تقریبی نمایش تصویر را در شرایط مختلف media به مرورگر اعلام می‌کند

مرورگر بر اساس این موارد بهترین تصویر را انتخاب می‌کند:

- نسبت پیکسلی دستگاه (DPR)
- عرض viewport
- فرمت‌های قابل پشتیبانی (مثل WebP)

چرا بهتر از media queryهای CSS برای تصویر است؟

- مرورگر قبل از دانلود تصمیم می‌گیرد (صرفه‌جویی در پهنای باند)
- درخواست HTTP اضافی برای تشخیص لازم نیست
- پشتیبانی از art direction (کراپ‌های متفاوت در breakpointهای مختلف)

مثال:

```html
<picture>
  <source media="(min-width: 800px)" srcset="large.jpg 1x, large-2x.jpg 2x" />
  <source media="(min-width: 400px)" srcset="medium.jpg" />
  <img
    src="small.jpg"
    srcset="small.jpg 400w, medium.jpg 800w"
    sizes="(max-width: 400px) 100vw, 800px"
    alt="..."
  />
</picture>
```

این روش مصرف دیتا را کم می‌کند و Core Web Vitals (به‌خصوص LCP) را بهبود می‌دهد.

## 🧠 سوال ۲۲

**آیدی**: html-022  
**عنوان**: اعتبارسنجی فرم در HTML5 چگونه کار می‌کند و با Constraint Validation API چطور می‌توان اعتبارسنجی سفارشی ساخت؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: فرم‌ها

### پاسخ 📄

HTML5 اعتبارسنجی داخلی فرم را با ویژگی‌هایی مثل `required`، `pattern`، `min/max` و بررسی نوع ورودی (مثل email و url) فراهم می‌کند.

فرآیند کلی:

- مرورگر هنگام submit یا focusout اعتبارسنجی می‌کند (با `novalidate` می‌توان غیرفعالش کرد)
- برای فیلد نامعتبر پیام خطای پیش‌فرض مرورگر نمایش داده می‌شود
- امکان افزودن خطای سفارشی با JavaScript وجود دارد

**Constraint Validation API**:

- دسترسی از طریق `element.validity` (شامل وضعیت‌هایی مثل `valueMissing` و `patternMismatch`)
- تعریف خطای سفارشی با `element.setCustomValidity('Error message')` (برای پاک کردن، رشته خالی)
- اجرای بررسی با `element.reportValidity()` یا `form.checkValidity()`

چرا این موضوع سخت است؟ چون موضوعاتی مثل i18n، اعتبارسنجی async (مثلاً یکتابودن username) و سازگاری مرورگرهای قدیمی مطرح می‌شوند.

مثال (اعتبارسنجی سفارشی دامنه ایمیل):

```html
<form>
  <input id="email" type="email" required pattern="[^@]+@example\.com" />
  <button type="submit">Submit</button>
</form>

<script>
  const email = document.getElementById('email');
  email.addEventListener('input', () => {
    if (email.value.endsWith('@gmail.com')) {
      email.setCustomValidity('Only @example.com allowed');
    } else {
      email.setCustomValidity('');
    }
  });
</script>
```

این API با ARIA هم برای دسترسی‌پذیری یکپارچه است (مثلاً `aria-invalid`).

- امنیت: باید ورودی sanitize شود تا XSS رخ ندهد

- افزودن ARIA مثل `role="textbox"` یا `role="region"`، `aria-label` و `aria-multiline="true"`
- مدیریت کیبورد: فوکوس، حرکت با کلیدها
- استفاده از `execCommand` یا Clipboard API مدرن برای قابلیت‌های ویرایش

مثال (ویرایشگر ساده):

```html
<div
  contenteditable="true"
  role="textbox"
  aria-multiline="true"
  aria-label="Editable area"
>
  Edit me!
</div>

<script>
  const editor = document.querySelector('[contenteditable]');
  editor.addEventListener('input', (e) => {
    // Sanitize: e.target.innerHTML = sanitize(e.target.innerHTML);
  });
</script>
```

برای محیط production معمولاً از کتابخانه‌هایی مثل Quill یا ProseMirror استفاده می‌شود.

## 🧠 سوال ۲۵

**آیدی**: html-025  
**عنوان**: Resource hintها در HTML (`preload`، `prefetch`، `preconnect`، `dns-prefetch`) چه هستند و هرکدام را چه زمانی برای بهینه‌سازی عملکرد استفاده می‌کنیم؟  
**سطح دشواری**: سخت  
**دسته‌بندی**: عملکرد

### پاسخ 📄

Resource hintها (در قالب `<link rel="...">`) به مرورگر می‌گویند منابع را زودتر resolve/fetch کند تا latency کم شود.

- **preload**: بارگذاری زودهنگام منابع حیاتی (مثل فونت یا اسکریپت ضروری). برای اولویت درست، `as="type"` مهم است.
- **prefetch**: بارگذاری کم‌اولویت و حدسی برای ناوبری بعدی (مثلاً منابع صفحه بعد).
- **preconnect**: برقراری زودهنگام اتصال TCP و handshake امنیتی برای دامنه‌های third-party.
- **dns-prefetch**: فقط resolve کردن DNS (سبک‌تر از preconnect، مناسب تعداد زیادی دامنه).

ترتیب هزینه تقریبی: dns-prefetch (کمترین) < preconnect < prefetch < preload (بیشترین اولویت).

مزایا: کاهش TTFB و بهبود FCP/LCP در Core Web Vitals.

نکته: استفاده بیش‌ازحد باعث هدررفت پهنای باند می‌شود؛ با Lighthouse تست کنید.

مثال:

```html
<!-- Preload critical font -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin />

<!-- Preconnect to CDN -->
<link rel="preconnect" href="https://cdn.example.com" crossorigin />

<!-- DNS prefetch for analytics -->
<link rel="dns-prefetch" href="https://analytics.com" />

<!-- Prefetch next page -->
<link rel="prefetch" href="/next-page.html" />
```

برای محتوای بالای صفحه (above-the-fold) از preload و برای مسیرهای ناوبری بعدی از prefetch استفاده کنید.

## 🧠 سوال ۲۶

**آیدی**: html-026
**عنوان**: چرا باید از `rel="noopener noreferrer"` روی لینک‌هایی با `target="_blank"` استفاده کرد؟
**سطح دشواری**: متوسط
**دسته‌بندی**: رفتار مرورگر و امنیت

### پاسخ 📄

وقتی یک لینک با `target="_blank"` در تب جدیدی باز می‌شود، صفحه جدید می‌تواند از طریق `window.opener` به صفحه اصلی دسترسی داشته باشد.

این دو خطر ایجاد می‌کند:

- **Tabnapping**: صفحه جدید می‌تواند تب اصلی را از طریق `window.opener.location` به یک صفحه فیشینگ هدایت کند
- **نشت Referrer**: هدر HTTP `Referer` آدرس صفحه مبدأ را به سایت مقصد فاش می‌کند

`rel="noopener"` — ارتباط `window.opener` را قطع می‌کند و از tabnapping جلوگیری می‌کند

`rel="noreferrer"` — علاوه بر قطع opener، ارسال هدر `Referer` را هم متوقف می‌کند (شامل `noopener` نیز می‌شود)

مرورگرهای مدرن به صورت خودکار `noopener` را برای `target="_blank"` اعمال می‌کنند، اما ذکر صریح آن همچنان بهترین روش است.

مثال:

```html
<!-- ریسک‌دار: دسترسی opener + نشت referrer -->
<a href="https://external.com" target="_blank">بازدید</a>

<!-- امن: بدون دسترسی opener، بدون نشت referrer -->
<a href="https://external.com" target="_blank" rel="noopener noreferrer"
  >بازدید</a
>
```

## 🧠 سوال ۲۷

**آیدی**: html-027
**عنوان**: تفاوت بین `<canvas>` و `<svg>` چیست؟
**سختی**: متوسط
**دسته‌بندی**: رفتار مرورگر

### پاسخ 📄

هر دو برای گرافیک استفاده می‌شوند اما به شکل متفاوتی کار می‌کنند.

**`<canvas>`**

- مبتنی بر رستر (رسم پیکسلی)
- از طریق JavaScript رندر می‌شود (API دستوری)
- اشکال رسم‌شده نود DOM ندارند
- مناسب برای رندر real-time، بازی‌ها و پردازش تصویر
- به طور پیش‌فرض دسترس‌پذیر نیست

**`<svg>`**

- مبتنی بر وکتور (اشکال ریاضی)
- نشانه‌گذاری declarative (مبتنی بر XML)
- هر شکل یک نود DOM است (قابل اسکریپت و استایل‌دهی)
- مناسب برای آیکون‌ها، نمودارها و تصاویر مقیاس‌پذیر
- دسترس‌پذیر و قابل ایندکس توسط موتورهای جستجو

مثال:

```html
<!-- Canvas: رسم دستوری -->
<canvas id="c" width="200" height="200"></canvas>
<script>
  const ctx = document.getElementById('c').getContext('2d');
  ctx.fillStyle = 'blue';
  ctx.fillRect(10, 10, 100, 100);
</script>

<!-- SVG: اشکال وکتوری declarative -->
<svg width="200" height="200">
  <rect x="10" y="10" width="100" height="100" fill="blue" />
</svg>
```

از `<canvas>` برای رندر پیکسلی داینامیک و از `<svg>` برای گرافیک مقیاس‌پذیر، تعاملی یا دسترس‌پذیر استفاده کنید.

### پاسخ 📄

تگ‌های Open Graph (OG) تگ‌های متایی هستند که نحوه نمایش صفحه را هنگام اشتراک‌گذاری در شبکه‌های اجتماعی مانند فیسبوک، توییتر/X، لینکدین و واتساپ کنترل می‌کنند.

این تگ‌ها ویژگی‌هایی مثل عنوان، توضیحات، تصویر و URL را برای کارت پیش‌نمایش لینک تعریف می‌کنند.

بدون تگ‌های OG، پلتفرم‌ها محتوا را از صفحه حدس می‌زنند و اغلب پیش‌نمایش ضعیفی تولید می‌کنند.

ویژگی‌های کلیدی OG:

- `og:title` — عنوان نمایش‌داده‌شده در پیش‌نمایش
- `og:description` — خلاصه نمایش‌داده‌شده در پیش‌نمایش
- `og:image` — آدرس تصویر بندانگشتی
- `og:url` — آدرس canonical صفحه
- `og:type` — نوع محتوا (website، article، video و ...)

مثال:

```html
<meta property="og:title" content="سوالات مصاحبه HTML" />
<meta
  property="og:description"
  content="راهنمای کامل آمادگی برای مصاحبه HTML."
/>
<meta property="og:image" content="https://example.com/preview.jpg" />
<meta property="og:url" content="https://example.com/html-questions" />
<meta property="og:type" content="website" />

<!-- مخصوص توییتر (اکنون X) -->
<meta name="twitter:card" content="summary_large_image" />
<meta name="twitter:title" content="سوالات مصاحبه HTML" />
```

تگ‌های OG با ارائه پیش‌نمایش‌های غنی و کنترل‌شده، نرخ کلیک را در شبکه‌های اجتماعی بهبود می‌دهند.
